# Testing Notes

## Payment Verification in Dev Mode

The Razorpay signature check in `POST /api/payment/verify` will **fail in Postman**
because the signature is generated by Razorpay's frontend SDK — it's not possible
to produce a valid one manually.

**For backend testing, temporarily disable the signature check:**

Open `backend/src/routes/payment.js`, find the verify route, and wrap the signature
check in an environment condition:

```javascript
// Verify HMAC signature
if (process.env.NODE_ENV === 'production') {
  const expectedSig = crypto
    .createHmac('sha256', process.env.RAZORPAY_KEY_SECRET)
    .update(`${razorpay_order_id}|${razorpay_payment_id}`)
    .digest('hex');

  if (expectedSig !== razorpay_signature) {
    logger.warn({ razorpay_order_id }, 'Payment signature mismatch');
    throw new AppError('Payment verification failed', 400, 'PAYMENT_SIGNATURE_MISMATCH');
  }
}
```

This skips signature check in `development` mode only.
**Never deploy to production with this change — revert before going live.**

After making this change, restart the backend:
```bash
make restart
```

Then the Postman payment verify request (step 3.2) will succeed.

## Audio/Infographic Generation Without Keys

Without ElevenLabs/Sarvam/Bannerbear keys, workers will fail and journey
won't auto-activate (since `checkAndActivateJourney` waits for all jobs).

**For testing without media keys**, you can manually activate a journey in the DB:

```bash
make shell-db
```

Then in psql:
```sql
-- Manually activate a journey (replace the UUID with your actual journey_id)
UPDATE journeys
SET status = 'active', calendar_started_at = NOW()
WHERE id = 'your-journey-id-here';

-- Create Day 1 session manually
INSERT INTO daily_sessions (journey_id, affirmation_day_number, calendar_date, state)
VALUES ('your-journey-id-here', 1, CURRENT_DATE, 'morning_unlocked');

-- Insert stub affirmation content for Day 1
INSERT INTO affirmation_days (journey_id, day_number, doubt, reframe, truth_statement, action_prompt, content_status)
VALUES (
  'your-journey-id-here',
  1,
  'I am not smart enough',
  'Every expert was once a beginner',
  'I am capable, I am learning, and I am enough exactly as I am today.',
  'Write down one thing you did well today, no matter how small.',
  'ready'
);
```

Now the journey route and check-in will work without any media generation.

## Checking Logs for Errors

When something doesn't work, always check the backend logs first:
```bash
make logs-backend
```

Common errors and what they mean:
- `"relation does not exist"` → migration hasn't run yet → `make migrate`
- `"invalid api key"` → wrong key in .env → check spelling, no extra spaces
- `"connection refused"` → Redis or Postgres not started → `make ps` to check
- `"cannot read properties of undefined"` → missing .env variable → check .env

## Viewing the Database

```bash
make shell-db
```

Useful queries:
```sql
-- See all users
SELECT id, name, email, subscription_tier, created_at FROM users;

-- See all journeys
SELECT id, user_id, track, status, affirmation_day, created_at FROM journeys;

-- See generated affirmation content
SELECT day_number, truth_statement, content_status FROM affirmation_days
WHERE journey_id = 'your-journey-id'
ORDER BY day_number;

-- See job queue status
SELECT job_type, status, COUNT(*) FROM content_generation_jobs
GROUP BY job_type, status;

-- See coaching messages
SELECT role, LEFT(content, 80) AS preview, created_at FROM coaching_messages
WHERE journey_id = 'your-journey-id'
ORDER BY created_at;
```
